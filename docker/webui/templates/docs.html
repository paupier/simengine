Simantha OPC UA Digital Twin

This system is a manufacturing digital twin — a real-time software model of a physical production line. It uses the Simantha discrete-event simulation library (NIST) to model machines, buffers, and material flow, then exposes everything through an OPC UA server so any SCADA, MES, or analytics tool can connect.

Unlike a batch simulator that runs once and stops, this system runs continuously at wall-clock speed — one simulation step per real second — making it suitable as a live digital twin that mirrors a physical line in real time.

Key capabilities

  • Serial production lines of 2–20+ machines connected by finite-capacity buffers
  • Realistic failure modeling via Weibull, exponential, and lognormal distributions
  • Multi-state health degradation with optional Condition-Based Maintenance (CBM)
  • Quality routing — defective parts are scrapped or sent for rework
  • SPC analytics — X-bar/R control charts, Cp/Cpk, Western Electric rules
  • Shift management — per-shift KPI resets and performance tracking
  • Event historian — structured event logs to CSV and/or InfluxDB
  • OPC UA interface — ISA-95/ISO 23247 aligned address space
  • Grafana dashboards via Telegraf → InfluxDB pipeline
  • Run recipes — ordered multi-product production schedules with changeovers

Buffer blocking and starvation

Buffers have finite capacity. When a buffer is full, the upstream machine cannot deposit a finished part and enters the BLOCKED state. When a buffer is empty, the downstream machine has no part to process and enters the STARVED state. Both conditions reduce throughput — a correctly sized buffer absorbs variability and keeps the line flowing.

Bottleneck rule: In a balanced line the bottleneck machine — the one with the longest cycle time — limits overall throughput. OEE is calculated using a bottleneck model: line OEE is the minimum-OEE machine's result.

The Per-Step Simulation Loop

The simulation runs at 1 step per real second. Each step calls system.simulate(simulation_time=1) — Simantha runs 1 second of discrete events then returns. This keeps simulation time locked to wall time and gives O(1) compute per step, regardless of how long the simulation has been running.

Reproducibility with --seed

The --seed CLI argument seeds both random and numpy.random before each step. Because the seed is base_seed + step_count, the same base seed always produces the same sequence of events — failure times, repair times, defect decisions, SPC noise — making runs fully reproducible for debugging or comparison.

If --seed is omitted, a seed is auto-generated from the current timestamp and printed to the console. Record it to reproduce interesting runs later.

Machine States

Every machine is in exactly one of seven states at any moment. States are detected by evaluating conditions in a fixed priority order after each simulation step.

IDLE
Machine is healthy, not blocked, not starved, and has no part currently in process. Waiting for the next part to arrive from the upstream buffer.
PROCESSING
A part is currently in the machine being worked on. The machine will finish after one cycle_time.
BLOCKED
Machine has finished a part but cannot deposit it because the downstream buffer is full. No production is occurring.
STARVED
Machine is ready to process but the upstream buffer is empty — no parts available to work on. Throughput loss.
DEGRADED
Machine health has dropped below 100% but has not yet failed. Still processing, but at increased defect rate if health_multiplier is configured. CBM triggers maintenance before failure.
FAILED
Health has reached the failed threshold (h_max). Machine is down, waiting for a maintainer to begin repairs. No production.
UNDER_REPAIR
Maintenance is in progress. The repair duration is sampled from the MTTR distribution. Machine returns to IDLE when repair_remaining reaches zero.

State detection order

Conditions are checked in this order — the first match wins:

if health >= failed_health:
    → FAILED (repair_remaining > 0 → UNDER_REPAIR)
elif machine.blocked:
    → BLOCKED
elif machine.starved:
    → STARVED
elif health > 0:
    → DEGRADED
elif machine.has_part:
    → PROCESSING
else:
    → IDLE

Warm-Up Period

When a simulation starts, the line is empty — buffers are empty, machines have no work in progress. It takes time for parts to fill the buffers and for the line to reach a steady state. Metrics collected during this transient phase are not representative of normal operation.

The warm_up_time parameter (in steps, i.e. seconds) tells the simulation how long to let the line stabilise before starting to count. During warm-up:

  • The simulation runs normally — machines process parts, failures can occur
  • OEE, PPM, quality counters, and downtime accumulators are not incremented
  • OPC UA KPI nodes remain at zero until warm-up completes
  • A console message confirms when warm-up ends and metrics become active
Common issue: If OEE/PPM are showing zero for an extended period after start, check warm_up_time in the scenario config. The full_feature_8_machine_line uses 120s warm-up by default.

Failure Modes & MTTF/MTTR

When enable_advanced_failures: true is set on a machine, it uses FailureModeManager to model realistic failure behaviour with distinct failure types that compete to cause the next failure (competing risks model).

MTTF — Mean Time To Failure

Defines how frequently a failure type occurs. Three distributions are supported:

DistributionTypical useParameters
weibull Wearout failures — hazard rate increases with age (e.g. mechanical fatigue) shape (β>1 = increasing hazard), scale (characteristic life)
exponential Random failures — constant hazard rate, memoryless (e.g. electrical) mean (MTTF)
lognormal General-purpose, models repair-time variability well mean, std

MTTR — Mean Time To Repair

Defines how long a repair takes. Also uses the three distributions above. MTTR is sampled once per failure at the moment the machine first reaches the failed state. The sampled duration is tracked in machine_repair_remaining, which counts down 1 second per step. The machine stays in UNDER_REPAIR until the countdown reaches zero.

Simple failures (no enable_advanced_failures) use scalar failure_rate and repair_time — Simantha's built-in exponential failure model with a fixed repair time.

Multiple concurrent failure modes

A single machine can have multiple failure modes (e.g., mechanical and electrical). The system uses a competing risks model: the next failure time is the minimum across all modes. When a failure occurs, the failing mode's MTTR distribution is used for the repair time.

Health Degradation (Multi-State)

Health degradation models gradual machine ageing using a discrete-state Markov chain. At each simulation step, the machine's health state can increase by 1 (degrade) with probability p_degrade, or stay the same.

Health state progression (h_max = 4 example)

Health state configuration

ParameterTypeDescription
h_maxintMaximum health state index. At this level the machine is FAILED. E.g. h_max=4 means 5 states (0–4).
p_degradefloatProbability per step of moving one state worse (0.01 = ~100 steps average in each state).
cbm_thresholdintHealth state at which the maintainer is called. If equal to h_max, machine must fail before repair (run-to-failure). If less than h_max, machine is repaired before failing (CBM).

CBM vs Run-to-Failure

Condition-Based Maintenance (CBM) triggers a repair before the machine fully fails. The key parameter is cbm_threshold relative to h_max:

CBM: cbm_threshold < h_max
Maintainer is called when health reaches the threshold. Machine is repaired before reaching h_max. Machine never enters FAILED state. Repair happens proactively during DEGRADED state. Best for high-value machines where failure is very costly.
Run-to-Failure: cbm_threshold == h_max
Machine must reach the fully-failed state before a maintainer is called. MTTR distribution is sampled at the moment of failure. Machine enters FAILED then UNDER_REPAIR. Appropriate for low-cost or easily replaced components.

MTTF scaling for multi-state machines

When a machine has multiple health states (h_max > 1), the sampled MTTF is divided by h_max to keep the total expected time-to-failure equal to the configured MTTF. Standard 2-state machines (h_max=1) are unaffected.

Quality Routing

When quality_routing.enabled: true is configured on a machine, defective parts are diverted away from the main line instead of passing through to the next station.

Routing modes

ModeBehaviour
scrap Defective parts are immediately sent to the configured scrap sink. They leave the line permanently and increment the scrap counter.
scrap_and_rework Defective parts first go through virtual rework (up to max_rework attempts). Each attempt succeeds with probability rework_success_rate. Parts that fail all rework attempts are scrapped.

Health-correlated defect rate

When enable_health_correlation: true, the effective defect rate is: 

effective_defect_rate = base_defect_rate × (1 + health_multiplier × health_state / h_max)

A machine with defect_rate: 0.05 and health_multiplier: 5 at health state 2 of 4 would have an effective defect rate of 5% × (1 + 5×0.5) = 17.5%.

Quality KPI counters

Quality routing increments per-machine counters: good_count, scrap_count, defective_count, rework_count, rework_success_count. These accumulate in LineState.MachineTotals across all simulation steps — the OPC UA QualityRouting/ nodes always show cumulative totals.

Quality counters are warm-up aware: they only increment after step_count ≥ warm_up_time. Physical routing (scrap redirect, part marking) still occurs during warm-up.

SPC Analytics

Statistical Process Control (SPC) monitors the cycle time of each configured machine for signs of process drift or special-cause variation. When enabled, the machine periodically samples its measured cycle time and adds it to an X-bar/R control chart.

Control chart mechanics

  • Subgroup: subgroup_size measurements are collected, then averaged (X-bar) and ranged (R)
  • Control limits: Calculated from the first num_subgroups subgroups. UCL/LCL are ±3σ from the grand mean.
  • Measurement noise: Gaussian noise with CoV = measurement_noise (default 2%) is applied to each measurement — models real-world gauge variation

Western Electric rules

When enable_western_electric: true, these out-of-control signals are detected:

RuleSignal
Rule 11 point beyond 3σ control limits
Rule 29 consecutive points on the same side of the centre line
Rule 36 consecutive points trending in the same direction
Rule 414 consecutive points alternating up-down

Capability indices

When usl and lsl (upper/lower specification limits) are configured:

  • Cp = (USL − LSL) / 6σ — process spread relative to specification width
  • Cpk = min(Cpu, Cpl) — measures how centred the process is within limits. Cpk ≥ 1.33 is generally considered capable.

OEE — Overall Equipment Effectiveness

OEE is the standard KPI for manufacturing efficiency. It is a composite metric combining three factors, each in range [0, 1]. OEE = 1.0 (100%) means the line produced only good parts, at full speed, with no downtime.

OEE = Availability × Performance × Quality
OEE
Availability (A)
How often the machine was running vs. total available time.
A = (shift_time - down_time) / shift_time
Performance (P)
How fast the machine ran vs. its theoretical maximum speed.
P = parts_made / (available_time / cycle_time)
Quality (Q)
What fraction of parts produced were good (no scrap).
Q = good_parts / total_parts_made

Per-machine vs line OEE

OEE is calculated individually for every machine using shift-relative time windows — counters reset at each shift boundary. Line-level OEE uses a bottleneck model: it takes the minimum OEE across all machines, reflecting that the weakest link constrains the whole line.

Warm-up aware: Availability uses metrics["down_time"] (our post-warm-up accumulator, not Simantha's). Performance uses machine.parts_made (post-warm-up only). Quality uses _good_count etc., which only increment after warm-up completes.

PPM / Throughput

Parts Per Minute (PPM) is the actual throughput of the line or individual machines. Two values are tracked:

  • TargetPPM — derived from cycle_time: 60 / cycle_time. The theoretical maximum rate for that machine.
  • ActualPPM — measured over the current shift: parts_made / elapsed_shift_time × 60.

Total throughput (Throughput OPC UA node) is the rate at which parts exit the Sink — good parts that completed the full line. It reflects the combined effect of all machine availabilities and the bottleneck constraint.

Shift Management

Shifts divide production time into named windows (e.g. Day, Evening, Night). When a shift boundary is crossed, per-shift metrics reset to zero — enabling shift-vs-shift comparison without losing cumulative totals.

Shift configuration

shifts:
  schedule:
    - name: "Day Shift"
      duration: 28800      # seconds (8 hours)
      start_offset: 0      # seconds from cycle start

    - name: "Evening Shift"
      duration: 28800
      start_offset: 28800  # starts after Day Shift

    - name: "Night Shift"
      duration: 28800
      start_offset: 57600  # starts after Evening Shift

The shift cycle repeats continuously. Shift time is sim-time based (not wall clock), so an 8-hour shift is 8×3600 = 28,800 simulation steps.

OPC UA nodes (ShiftManagement/)

  • CurrentShiftName — name of active shift
  • ShiftNumber — 1-based index within the cycle
  • ShiftElapsed, ShiftRemaining — seconds
  • ShiftPartsProduced, ShiftGoodParts
  • ShiftOEE — OEE over the current shift only

Maintenance Strategies

When a maintainer is enabled (maintainer.enabled: true), the PriorityMaintainer manages repairs. If multiple machines fail simultaneously, the repair scheduling strategy determines which is fixed first.

StrategyBehaviour
fifoFirst-in, first-out. Machines are repaired in the order they failed. Default strategy.
sptShortest Processing Time. Machine with the shortest MTTR estimate is repaired first — minimises average downtime per repair.
priorityUser-assigned numeric priorities (machine_priorities dict). Lower number = higher priority. Ties broken by FIFO.
bottleneckMachine with the longest cycle time (lowest throughput) is repaired first — maximises line throughput recovery.

Maintainer capacity

maintainer.capacity sets how many machines can be under repair simultaneously. The default capacity is 1 (one repair team). The 8-machine scenario uses capacity 2.

Storage backends

Docker deployment: inject_historian.py runs at container start and enables InfluxDB for all scenarios, pointing at the Docker-internal InfluxDB URL. Your warm_up_time and other edits in the runtime config are preserved across restarts.

Run IDs

Every simulation run gets a unique Run ID in the format {scenario}_{YYYYMMDD_HHMMSS}. This ID is propagated through every layer of the stack so you can isolate data from a specific run.

  • OPC UA: Identification/RunID variable
  • Telegraf: [global_tags] block — all InfluxDB points carry run_id tag
  • CSV historian: run_id column in every row
  • InfluxDB historian: run_id tag on every Point
  • Grafana dashboards: A run_id template variable lets you filter all panels to a specific run
  • Web UI → Run Reports: Lists all past runs; click to copy run_id for Grafana filter

Use the Run Reports page to see past run IDs. Click any ID to copy it, then paste into the Grafana run_id variable selector.

Reading the Line Balance dashboard

The Blocked vs Starved Time panel is the primary bottleneck locator. A machine with high BlockedTime (orange) cannot move parts forward because the next machine is too slow — the bottleneck is downstream. A machine with high StarvedTime (blue) is waiting for parts because the previous machine is too slow — the bottleneck is upstream. Corroborate with the Buffer Levels panel: a buffer that stays full confirms its downstream machine is the constraint.

Generating MTTR data

The Downtime & Reliability dashboard requires machines to reach the FAILED state. With CBM enabled (default on most scenarios), machines are serviced before failing and the FAILED / UNDER_REPAIR states may rarely appear. To generate clear failure and repair data, run the full_feature_8_machine_line_rtf scenario — it disables CBM so all machines run to failure, producing frequent FAILED → UNDER_REPAIR → PROCESSING transitions and realistic MTTR readings.

Run Recipes

Recipes define ordered multi-segment production schedules. Each segment runs against a base scenario with optional per-machine overrides, separated by changeover periods. Use recipes to model multi-product scheduling, shift schedules, or product qualification runs.

Recipe structure

name: "Monday Production Schedule"
base_scenario: full_feature_line   # which scenario topology to use

segments:
  - name: "Product A Morning"
    quantity: 500          # stop after 500 good parts
    max_duration: 18000    # safety timeout (5h)
    overrides:
      machines:
        - name: M1
          cycle_time: 10   # override cycle time for this product
          defect_rate: 0.02
    changeover:
      target: 300          # planned changeover duration (seconds)
      distribution: lognormal
      mean: 300            # actual duration is sampled from this distribution
      std: 60

  - name: "Product B Afternoon"
    duration: 7200         # time-box: run for exactly 2h
    overrides: ...

Stop conditions

FieldTypeBehaviour
quantityintStop when this many good parts have been produced in the segment
durationint (s)Stop after this many seconds regardless of quantity
max_durationint (s)Safety timeout when using quantity stop — prevents infinite runs if throughput is very low

Changeover distributions

Changeover durations can be deterministic or stochastic, modelling real-world changeover variability:

  • constant — fixed value (deterministic)
  • normal — Gaussian with mean and std
  • lognormal — log-normal (always positive), good for changeover times
  • exponential — exponential with mean

During changeover the OPC UA OperationsState/Recipe/ChangeoverState node is set to "CHANGEOVER" and no parts are produced.

OPC UA Address Space

The OPC UA address space follows the ISA-95 / IEC 62264 equipment hierarchy aligned with ISO 23247 digital twin identification. All nodes are read-only during a run. 

Enterprise (WeylandIndustries)
└── Site (LV426_Colony)
    └── Area (AtmosphereProcessor01)
        └── {line_name}_Equipment
            ├── Identification/       EquipmentID, EquipmentClass, Description, RunID
            ├── OperationsState/      SimTime, LineState, LineMode
            │   └── Controls/         SetInterarrivalTime (read-only)
            │   └── Recipe/           RecipeName, SegmentName, SegmentIndex, …
            ├── OperationsPerformance/ Throughput, TotalWIP, TotalScrap, ScrapRate
            ├── OEE/                  Availability, Performance, Quality, OEE, …
            └── Resources/
                ├── M{i}_Equipment/
                │   ├── OperationsState/   State, HealthState, HealthPercent
                │   ├── OperationsPerformance/ PartCount, Utilisation, TargetPPM, ActualPPM
                │   ├── OEE/               Availability, Performance, Quality, OEE
                │   ├── Alarms/            MachineFailureActive, MaintenanceActive, QualityAlertActive
                │   ├── SPC/               (if enable_spc)
                │   └── QualityRouting/    GoodCount, ScrapCount, ReworkCount, … (if quality_routing)
                ├── B{i}_StorageUnit/      CurrentLevel, Capacity, Alarms/
                └── SupportFunctions/
                    ├── Maintenance/       MaintenanceActive, QueueLength, TotalRepairs
                    └── ShiftManagement/   (if shifts configured)

Connecting via OPC UA

Endpoint: opc.tcp://localhost:4840/simantha/ (no authentication). Use any OPC UA client (UaExpert, Python asyncua, etc.) to browse and subscribe to nodes. Telegraf polls all nodes automatically when running in Docker.

Glossary

ActualPPM
Measured parts-per-minute rate for a machine during the current shift. Contrast with TargetPPM.
Availability (A)
OEE factor: fraction of planned production time that the machine was actually running (not failed or in repair). A = (shift_time − down_time) / shift_time.
BLOCKED
Machine state: machine has completed a part but cannot move it downstream because the output buffer is full.
Buffer
A finite-capacity queue between two machines. Decouples machine speeds and absorbs variability. When full, the upstream machine is BLOCKED. When empty, the downstream machine is STARVED.
CBM
Condition-Based Maintenance. Maintenance triggered by machine health reaching a threshold (cbm_threshold < h_max) before actual failure. Prevents FAILED state.
Changeover
The period between recipe segments during which the line is reconfigured for a new product. Duration is sampled from a distribution. No parts are produced.
Cp
Process capability index. Cp = (USL − LSL) / 6σ. Measures how well the process spread fits within the specification limits. Cp ≥ 1.33 is typically required.
Cpk
Process capability index accounting for centring. Cpk = min((USL−μ)/3σ, (μ−LSL)/3σ). Lower than Cp indicates the process mean is off-centre.
cycle_time
Seconds per part for one machine. The reciprocal of throughput rate. target_ppm overrides this: cycle_time = 60 / target_ppm.
defect_rate
Probability [0,1] that any given part exiting a machine is classified as defective. Combined with health_multiplier for health-correlated quality.
DEGRADED
Machine state: health > 0 but below the failure threshold. Machine is still operating but at potentially higher defect rate. CBM triggers repair during this state.
Digital Twin
A real-time virtual model of a physical asset or system that mirrors its current state and behaviour. This system is a manufacturing line digital twin.
EventHistorian
The event logging subsystem. Detects state transitions and records structured events to CSV and/or InfluxDB. Abstract base class with CSV, InfluxDB, and Neo4j backends.
FAILED
Machine state: health has reached h_max. Machine is down and not producing. Waiting for a maintainer to begin repair.
failure_rate
Simple (non-advanced) failure parameter: failures per hour, modelled as Poisson process. Replaced by failure_modes when using advanced failures.
Grafana
Open-source visualisation platform. Pre-provisioned dashboards query InfluxDB via Flux queries, filtered by run_id for per-run analysis.
health_multiplier
Scaling factor for health-correlated quality. Effective defect rate = base_defect_rate × (1 + health_multiplier × health_fraction).
health_states
Multi-state degradation config block: h_max (failed state index), p_degrade (per-step degrade probability), cbm_threshold.
historian
Config block enabling the EventHistorian for a scenario. Can enable CSV, InfluxDB, and Neo4j backends independently with per-event-type control.
h_max
Maximum health state index. Machine is FAILED at this level. Total health states = h_max + 1 (states 0 through h_max).
IDLE
Machine state: healthy, no part being processed, not blocked, not starved. Waiting for a part to arrive.
InfluxDB
Time-series database. Stores Telegraf metrics (sim_metrics measurement) and historian events (sim_events measurement) tagged by run_id.
ISA-95
International standard (IEC 62264) for enterprise-control system integration. Defines a 4-level hierarchy: Enterprise → Site → Area → Work Centre. The OPC UA address space follows this model.
LineState
Python class that accumulates per-step simulation deltas into monotonically increasing counters. Survives simulate() reinitialisations. Also the OPC UA variable name for the line operating mode.
lognormal
Probability distribution: always positive, right-skewed. Commonly used for repair time modelling. Specified by mean and std of the underlying normal distribution.
MachineTotals
Dataclass holding cumulative counters per machine (parts_made, good_count, scrap_count, defective_count, rework_count, rework_success_count). Populated by LineState.sync_machine().
MTTF
Mean Time To Failure. Expected time between failures for a given failure mode. Specified as a probability distribution (Weibull, exponential, lognormal).
MTTR
Mean Time To Repair. Expected duration of one repair event. Sampled from a distribution at the moment of failure. Tracked via machine_repair_remaining countdown.
OEE
Overall Equipment Effectiveness = Availability × Performance × Quality. Industry-standard composite KPI for manufacturing efficiency. 1.0 = perfect production.
OPC UA
OPC Unified Architecture. Industrial communication protocol (IEC 62541). Platform-independent, secure, with a rich information model. Endpoint: opc.tcp://localhost:4840/simantha/
p_degrade
Per-step probability of the machine degrading one health state. p_degrade=0.01 means on average 100 steps per health state.
Performance (P)
OEE factor: actual parts produced vs. theoretical maximum at full speed. P = parts_made / (available_time / cycle_time).
PPM
Parts Per Minute. Throughput metric. TargetPPM = 60/cycle_time. ActualPPM = parts_made / elapsed_minutes during current shift.
PROCESSING
Machine state: a part is currently in the machine being worked on, completing after one cycle_time.
Quality (Q)
OEE factor: good parts as a fraction of total parts produced. Q = good_parts / total_parts_made.
quality_routing
Per-machine config block enabling defect diversion to scrap sinks or rework. Modes: scrap, scrap_and_rework.
Recipe
A YAML file defining an ordered sequence of production segments on a base scenario. Each segment can override machine parameters and is separated by a changeover period.
repair_time
Fixed repair duration for simple (non-advanced) failures. Replaced by MTTR distribution when using advanced failures.
rework_success_rate
Probability [0,1] that a single rework attempt successfully salvages a defective part. Parts failing all max_rework attempts are scrapped.
run_id
Unique identifier for one simulation run: {scenario}_{YYYYMMDD_HHMMSS}. Tag on all InfluxDB data points and CSV rows. Used to isolate and filter data from a specific run.
Scenario
A named line configuration in line_models.yaml defining machines, buffers, failures, quality, shifts, historian, and maintainer for one simulation run.
scrap_sink
A named collection point for defective parts. Parts diverted to a scrap sink leave the production line permanently and are counted in TotalScrap.
Simantha
NIST open-source Python library for discrete-event manufacturing simulation. Provides Machine, Buffer, Source, Sink, Maintainer primitives. This system wraps Simantha with OPC UA, SPC, quality routing, and historian layers.
Sink
Simantha object representing the end of the line. Receives good parts that have passed all machines. sink.level is the per-step part count used to track throughput.
SPC
Statistical Process Control. Uses X-bar/R control charts to monitor process stability. Detects special-cause variation via Western Electric rules. Calculates Cp/Cpk capability.
STARVED
Machine state: ready to process but the upstream buffer is empty. No parts available. Causes throughput loss.
target_ppm
Alternatively specify machine speed as parts-per-minute instead of cycle_time. Derives cycle_time = 60 / target_ppm. Takes precedence if both are given.
Telegraf
InfluxData metrics collection agent. Configured to poll OPC UA nodes every second and write to InfluxDB. Config auto-generated per scenario by generate_telegraf_conf.py.
UNDER_REPAIR
Machine state: repair is in progress. machine_repair_remaining counts down from the sampled MTTR. Machine returns to IDLE when countdown reaches zero.
warm_up_time
Number of simulation steps (seconds) before KPI collection begins. During warm-up the line fills to steady state but metrics are not incremented.
Weibull
Probability distribution with shape parameter β. β>1 = increasing failure rate (wearout). β=1 = exponential (random). β<1 = decreasing (infant mortality). Used for MTTF modelling.
Western Electric rules
Set of out-of-control detection rules for SPC control charts. Four rules detect patterns suggesting the process mean or variance has shifted. Triggers SPC_VIOLATION events.
WIP
Work In Progress. Total parts currently in the system (in machines + in buffers). High WIP can indicate a bottleneck. Tracked in TotalWIP OPC UA variable.
X-bar / R chart
Control chart pair: X-bar monitors subgroup means (process average), R monitors subgroup ranges (process variability). Both must be in control for a stable process.
